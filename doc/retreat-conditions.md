# 撤退条件・成功指標

## 概要

「やめる条件」と「勝ちの定義」を、プロジェクト開始前に決める。
感情ではなく数字で判断するための仕組み。

---

## 成功指標

**プロジェクト開始前に記入する。後出しで変更しない。**

| 指標 | 目標値 | 測定方法 | 判定時期 |
|------|--------|---------|---------|
| | | | |

### 成功指標の決め方

- 具体的で測定可能な数字にする（「良くなる」ではなく「20%改善」）
- ベースライン（現状値）を先に測っておく
- 判定時期を決める（いつ測るか）

---

## 撤退条件

### ソフトストップ（警告 → 人間が判断）

| 条件 | 閾値 | チェック方法 | チェック頻度 | アクション |
|------|------|------------|-----------|-----------|
| | | 手動 / 自動 | | 警告 → 判断 |

### ハードストップ（即停止）

| 条件 | アクション |
|------|-----------|
| データ不整合・バグ | 即停止 |
| セキュリティ違反 | 即停止 |
| コンプライアンス違反 | 即停止 |

---

## shadow降格条件

本番運用（live）から観察運用（shadow）に戻す条件。
撤退ではないが、本番からは外す。

| 条件 | 閾値 | アクション |
|------|------|-----------|
| | | shadow に戻して観察継続 |

---

## 判定ルール

- 撤退条件は**プロジェクト開始前に決める**
- 後から条件を緩めない（緩めたくなったら、それは感情）
- 自動チェックを組み込む（人間は忘れる）
- 撤退 ≠ 失敗。「この条件下では機能しなかった」という学びを記録する
